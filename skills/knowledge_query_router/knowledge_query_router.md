# 适用范围

- 当用户提出的问题可能与知识库（`data/kg_graph/`）中的内容相关时
- 当需要决定是查知识库、搜索网络还是直接回答时
- 当需要将多个信息源（KB、搜索、LLM 知识）综合为一份回答时

# 核心原则

1. **KB 优先但不盲从** — 先查知识库，但评估后决定是否补充搜索
2. **标注来源** — 明确告诉用户信息来自哪里（KB / 搜索 / 通用知识）
3. **时效感知** — 根据问题类型判断是否需要最新信息
4. **KB 建设意识** — 发现 KB 缺失的内容时标记为候选

# 决策流程

## 三阶段决策

```
用户问题
    │
    ▼
┌─────────────────────────────┐
│  阶段1：知识库检索            │
│  在 data/kg_graph/ 下搜索     │
│  相关页面                    │
└──────────┬──────────────────┘
           │
    找到相关内容？
    ├─ YES ──→ 阶段2：内容评估
    │           │
    │     KB 内容精确且最新？
    │     ├─ YES ──→ KB Only 模式回答
    │     └─ NO  ──→ 阶段3：补充搜索
    │                  ↓
    │              KB + Search 模式回答
    │
    └─ NO  ──→ 阶段2：意图分析
               │
         问题类型？
         ├─ 最新事件/实时数据 ──→ 搜索 ──→ Search Only
         ├─ 技术趋势/最新框架 ──→ 搜索 ──→ KB 通用原则 + Search
         ├─ 通用知识/原理 ─────→ 直接回答 ──→ Direct
         └─ 与已有 KB 主题相关 ──→ 搜索并标记 KB 候选
```

## 阶段 1：知识库检索

### 搜索策略（按优先级）

1. **文件名匹配** — 在 `data/kg_graph/` 下搜索与问题关键词相关的 `.md` 文件名
2. **Frontmatter 匹配** — 检查页面的 `tags` 和 `topic` 字段
3. **全文内容匹配** — 搜索页面内容中与问题相关的段落

### 搜索范围

```
data/kg_graph/
  ├── INDEX.md          ← 先看总索引，了解有哪些知识库
  ├── <topic>/
  │   ├── INDEX.md      ← 看该主题的页面列表
  │   ├── glossary.md   ← 术语表
  │   └── pages/        ← 具体知识页面
  │       └── *.md
  └── ...
```

### 操作

```
# 步骤1：查看有哪些知识库
读取 data/kg_graph/INDEX.md

# 步骤2：如果问题与某个知识库主题相关，读取该知识库的 INDEX.md
读取 data/kg_graph/<topic>/INDEX.md

# 步骤3：读取最相关的页面
读取 data/kg_graph/<topic>/pages/<page>.md

# 步骤4：如果不确定哪个页面相关，搜索关键词
在 data/kg_graph/ 下搜索包含关键词的文件
```

## 阶段 2：内容评估 / 意图分析

### 当 KB 有内容时 — 三个评估维度

| 维度 | 检查项 | 判断标准 |
|------|--------|----------|
| **精确度** | 内容是否直接回答用户的具体问题？ | 内容覆盖了用户问题的核心，而非仅相关 |
| **时效性** | 页面 `updated` 日期是否合理？ | 时效性问题：3个月内；一般知识：1年内可接受 |
| **完整性** | 是否覆盖了用户问题的所有方面？ | 如果用户问了多个方面，KB 是否都涉及 |

### 当 KB 无内容时 — 意图分类

| 问题类型 | 示例 | 是否需要搜索 | 原因 |
|---------|------|------------|------|
| **最新事件/新闻** | "今天有什么科技新闻？" | **必须** | KB 不可能有 |
| **实时数据** | "明天北京天气？"、"现在比特币价格？" | **必须** | 动态变化 |
| **技术趋势/最新框架** | "2026 年最好的 Agent 框架？" | **建议** | 更新快，KB 可能过时 |
| **通用概念/原理** | "什么是函数式编程？" | **不必** | LLM 知识足够 |
| **操作步骤/最佳实践** | "如何用 Python 做异步编程？" | **视情况** | 可能变化，但基础不变 |
| **个人观点/偏好** | "你推荐学什么语言？" | **不必** | 主观问题 |

## 阶段 3：执行与综合

### 四种回答模式

| 模式 | 触发条件 | 行为 | 来源标注 |
|------|---------|------|----------|
| **KB Only** | KB 内容精确且最新 | 直接引用 KB 内容回答 | "根据知识库中的 [页面名]" |
| **KB + Search** | KB 有基础但需补充 | KB 作为基础，搜索补充新信息 | "知识库中提到...，最新的补充..." |
| **Search Only** | KB 无相关内容且需要最新信息 | 搜索后回答 | "根据最新信息..." |
| **Direct** | KB 无相关内容且问题为通用知识 | 直接利用 LLM 知识回答 | 无需特别标注 |

### 各模式的执行流程

#### KB Only 模式

```
1. 读取相关 KB 页面
2. 提取与用户问题直接相关的内容
3. 用自然语言组织回答
4. 标注来源：「根据知识库中的 [页面名]」
5. 如果 KB 内容不足以完全回答，说明还缺少哪些信息
```

#### KB + Search 模式

```
1. 读取相关 KB 页面，提取基础内容
2. 搜索最新/补充信息
3. 综合两者：KB 作为基础框架，搜索结果作为补充
4. 明确区分哪些来自 KB、哪些来自搜索
5. 如果有冲突，说明差异并给出判断
```

#### Search Only 模式

```
1. 分析问题类型，确定搜索关键词
2. 执行搜索
3. 整理搜索结果，组织回答
4. 标记：「这是最新搜索的结果」
5. 如果此内容与某个 KB 主题相关，标记为 KB 候选
```

#### Direct 模式

```
1. 判断问题为通用知识类
2. 利用 LLM 自身知识直接回答
3. 如果回答涉及操作或可能有变化，提醒用户"如需最新信息可以搜索"
```

# 输出规范

## 回答格式

### 使用 KB 内容时

```
根据知识库中的信息：

[KB 内容的精炼总结，不是直接复制]

> 来源：知识库 — [页面标题] (data/kg_graph/<topic>/pages/<page>.md)
```

### 使用 KB + 搜索时

```
关于这个问题，知识库中有一些基础内容，我补充了最新的信息：

**基础知识（来自知识库）**：
[KB 内容]

**最新补充（来自搜索）**：
[搜索结果]

> 知识库来源：[页面标题]
```

### 仅搜索时

```
根据最新信息：

[搜索整理后的回答]

> 注：知识库中暂无相关内容，如需我可以整理到知识库。
```

### 直接回答时

```
[直接回答]

> 注：这是我的通用知识，如果需要验证或补充最新信息，可以告诉我。
```

## 回答原则

- **先给结论，再给依据** — 用户先看答案，再看来源
- **不要直接复制 KB 内容** — 提炼总结，用自然语言
- **标注来源要简洁** — 不要像学术论文那样冗长
- **多来源冲突时说明** — "知识库说 A，但最新信息显示 B"

# 知识库建设反馈

当在回答问题过程中发现知识库缺失或不足时：

| 情况 | 行动 |
|------|------|
| KB 完全无相关内容 | 标记为"KB 候选"，建议用户是否需要添加 |
| KB 有但内容过时 | 标记为"需更新"，建议用户是否需要更新 |
| KB 有但不够完整 | 标记为"需补充"，建议用户是否需要完善 |
| 搜索到高质量内容 | 建议用户是否需要导入到 KB |

> **注意**：不要自动修改 KB 内容。只有在用户明确要求时才执行 KB 更新操作。

# 注意事项

- 搜索和读取 KB 是两个独立的操作，不要混淆
- KB 内容视为已验证信息，搜索内容需要自己判断可信度
- 时效性问题（天气、价格、新闻）不要依赖 KB
- 对于用户明确问"知识库里有没有..."的问题，只查 KB，不搜索
- 回答简洁，来源标注放在最后，不要过度强调
