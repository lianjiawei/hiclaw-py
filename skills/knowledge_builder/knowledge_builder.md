# 适用范围

- 当用户提到构建/创建知识库、知识管理、整理知识、建立 Wiki 时使用
- 当用户要求按主题分类组织知识时使用
- 当用户需要导入内容到知识库、查询已有知识时使用
- 当用户要求更新、维护或整理现有知识库时使用

# 核心原则（Karpathy LLM Wiki）

知识库按以下原则构建，确保 Agent 和用户都能高效使用：

## 1. 结构清晰，层次扁平

- 知识库采用"扁平为主、层次为辅"的结构
- 避免超过 3 级深度的嵌套目录
- 根目录下只放 `INDEX.md`（总索引）和少数几个主题子目录
- 每个子目录代表一个明确的知识领域

## 2. 文件名即搜索词

- 文件名使用**描述性、可搜索**的名称，用 `-` 连接单词
- 例如：`python-async-programming.md` 而非 `note1.md`
- 避免缩写和含糊名称
- 文件名应直接反映页面内容

## 3. 页面自包含 + 交叉链接

- 每个页面应尽可能**自包含**，阅读时无需跳转多处
- 同时在页面底部维护 `## Related` 区域，列出双向链接
- 链接使用相对路径，例如 `[异步编程](python-async-programming.md)`
- 被引用的概念要有自己的页面

## 4. 结构化写作，LLM 友好

- 使用清晰的 Markdown 标题层级（H1 > H2 > H3）
- 每个概念段落以**定义句**开头
- 使用表格、列表、代码块等结构化元素
- 避免含糊表述，用**具体事实**替代模糊描述
- 问题-解决格式优先

## 5. 唯一真相源（Single Source of Truth）

- 每条信息只在一个地方定义
- 避免在不同页面重复同一信息
- 需要复用时，通过链接引用而非复制粘贴

## 6. 元数据前置

- 每个页面使用 YAML frontmatter 包含元数据：
  ```yaml
  ---
  title: 页面标题
  topic: 所属主题
  tags: [标签1, 标签2]
  created: YYYY-MM-DD
  updated: YYYY-MM-DD
  status: draft | stable | deprecated
  ---
  ```

## 7. 术语统一

- 维护一个全局 `glossary.md`，定义所有关键术语
- 术语首次出现时给出定义并链接到 glossary
- 避免同义词混用，选定一个标准术语

## 8. 渐进式细化

- 高层页面给出概要和导航
- 深层页面给出详细解释
- 读者可以从概览逐步深入到细节

# 知识库目录结构

知识库默认存储在 `data/kg_graph/` 下，按主题组织：

```
data/kg_graph/
  <topic-name>/
    INDEX.md              ← 该主题的总索引和概览
    glossary.md           ← 术语表
    pages/                ← 具体知识页面
      <page-slug>.md
      ...
    templates/            ← 该主题下的自定义模板（可选）
    archive/              ← 归档页面（可选）
```

多个主题并存：

```
data/kg_graph/
  machine-learning/
  web-development/
  finance/
  ...
```

# 工作目标

1. 识别用户想构建的知识主题
2. 在 `data/kg_graph/` 下创建结构化的知识库
3. 将内容按原则导入知识库
4. 支持后续更新、查询、维护知识库
5. 维护知识库健康（链接、重复、过期内容）

# 执行步骤

## 阶段 1：创建知识库

1. 确定知识库的**主题名称**（英文小写，`-` 连接）
2. 在 `data/kg_graph/<topic-name>/` 下创建目录结构：
   - `INDEX.md` — 主题概览 + 导航
   - `glossary.md` — 术语表
   - `pages/` — 知识页面目录
3. 初始化 `INDEX.md`，包含：
   - 主题简介
   - 页面列表（链接）
   - 标签索引
   - 维护状态
4. 初始化 `glossary.md`，包含关键术语占位符

## 阶段 2：导入内容

1. 分析用户提供的内容（文件、文本、链接等）
2. 判断内容应归属哪个已有知识库（或创建新库）
3. 将内容转换为 LLM Wiki 格式：
   - 添加 frontmatter
   - 拆分过长页面（> 500 行建议拆分）
   - 提取关键术语更新 glossary
   - 添加 Related 链接
4. 存入 `pages/` 目录，文件名为描述性 slug
5. 更新 `INDEX.md` 的页面列表和标签索引

## 阶段 3：更新知识库

1. 用户指定要更新的内容或页面
2. 定位现有页面
3. 遵循以下更新原则：
   - 保留 frontmatter，更新 `updated` 时间
   - 合并新旧信息，消除冲突
   - 更新 Related 链接
   - 如涉及术语变更，同步更新 `glossary.md`
4. 记录变更摘要

## 阶段 4：查询知识库

1. 用户提出问题或指定关键词
2. 在 `data/kg_graph/` 下搜索相关页面
3. 优先匹配文件名和 frontmatter tags
4. 返回匹配页面的核心内容摘要
5. 如未找到，明确告知并建议是否要添加

## 阶段 5：维护健康

定期检查并执行：
- 检查 broken links（指向不存在文件的链接）
- 检测内容重复（相似页面合并建议）
- 更新过期页面状态（stable → deprecated）
- 更新 `INDEX.md` 索引一致性

# 输出要求

- 创建知识库时，展示创建的目录结构和初始化内容
- 导入内容时，说明导入到哪个页面、拆分了哪些内容
- 更新内容时，列出变更摘要
- 查询时，给出匹配结果和简要内容，附上页面路径
- 维护时，报告发现的问题和建议修复

# 注意事项

- 主题名称使用英文小写 + 连字符（如 `machine-learning`）
- 页面文件名同样使用英文小写 + 连字符
- 不要跨知识库主题混放文件
- 归档页面移到 `archive/` 而非删除
- 保持页面大小合理（建议 < 500 行/页）
- frontmatter 中的 `status` 反映页面可信度
- 中文内容可以写在页面正文中，但文件名和目录名用英文

# 页面模板

新建页面时使用以下模板结构：

```markdown
---
title: [页面标题]
topic: [所属主题]
tags: [标签1, 标签2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: draft
---

# [页面标题]

## Overview

[1-2 句话说明本页面的核心内容]

## Key Concepts

### 概念 1

[定义句] 具体解释、示例或代码。

### 概念 2

[定义句] 具体解释、示例或代码。

## Details

[详细展开的内容，按问题-解决或分类组织]

## Examples

[具体示例，代码块、表格或步骤说明]

## Common Pitfalls

- 常见错误 1：说明和避免方法
- 常见错误 2：说明和避免方法

## Related

- [相关页面 1](./相关页面-slug.md)
- [相关页面 2](../其他主题/相关页面-slug.md)
```
